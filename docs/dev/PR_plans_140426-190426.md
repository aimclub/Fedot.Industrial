# План PR на 140426-190426

Стратегия строится не “по слоям кода”, а по снижению архитектурного риска. Сейчас у нас уже есть хороший фундамент,
поэтому следующий цикл лучше вести так, чтобы каждый PR уменьшал скрытую связность и одновременно повышал ценность для
benchmark/runtime.

## Планируемый порядок

1. **PR-1 Legacy Forecasting Boundary Cleanup**
   Самый дешёвый по риску и сразу уменьшает вероятность, что новый forecasting runtime всё ещё тайно зависит от старого
   industrial_*_strategy.
   После него база для всего остального буде чище.

2. PR-2 OKHS Primitive-Orchestration Extraction
   Самый важный архитектурный PR следующего цикла. OKHS всё ещё остаётся главным “остаточным монолитом”

3. PR-4 Repository and Search-Space Consolidation
   Я бы поднял этот PR раньше PR-3. Причина простая: если tuning делать до полной нормализации
   vocabulary/defaults/search-space, мы рискуем закрепить не тот public contract. Сначала выравниваем
   names/defaults/aliases, потом уже строим tuning orchestration поверх стабильного словаря.

3. PR-3 Explicit Stage-Tuning Contract
   После cleanup + OKHS extraction + repository consolidation можно уже спокойно вводить stage-aware tuning:

trajectory stage
decomposition/rank stage
forecast head stage
Это даст уже не косметический, а реально полезный инженерный эффект.
PR-5 Benchmark Stage Adoption Completion
На этом шаге benchmark станет окончательным “зеркалом архитектуры”, а не просто местом запуска моделей. Я бы делал этот
PR после tuning contract, чтобы benchmark сразу умел показывать stage-level behavior уже в зрелом виде.

# PR-6 Neural Forecast Head Bridge

Это я бы оставил последним. Он важный, но только после того, как non-neural primitive graph будет окончательно стабилен.

Почему именно так
Сейчас главный риск не в том, что “чего-то не хватает”, а в том, что часть системы уже живёт в новой архитектуре, а
часть ещё может незаметно возвращать нас в legacy semantics. Поэтому стратегия должна быть:

сначала убрать скрытую связность;
потом стабилизировать source-of-truth;
потом строить tuning/runtime orchestration;
и только после этого расширять систему дальше.
Практический план цикла
Я бы делил следующий цикл на 3 блока:

Блок A: PR-1 + PR-2
Цель: сделать runtime и OKHS действительно shell-first.
Блок B: PR-4 + PR-3
Цель: зафиксировать vocabulary и поверх него ввести stage-aware tuning.
Блок C: PR-5 + PR-6
Цель: завершить benchmark adoption и подготовить neural bridge.
Если выбирать один лучший следующий PR, я бы взял первым именно PR-1 Legacy Forecasting Boundary Cleanup, а сразу за ним
PR-2 OKHS Primitive-Orchestration Extraction. Это даст максимальный архитектурный выигрыш на единицу усилия.

Если хотите, следующим сообщением я могу уже разложить PR-1 в очень конкретный implementation plan:

какие файлы менять,
какие invariants сохранить,
какие тесты добавить,
и какой будет acceptance checklist.
