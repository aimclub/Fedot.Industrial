# Обзор KL-изменений перед вливанием в `main`

Документ описывает состояние IndustrialTS после закрытия серии задач с префиксом `KL`. Он нужен внешним разработчикам и новым участникам проекта: чтобы быстро понять, что изменилось в репозитории, где теперь находятся основные точки входа и каким правилам стоит следовать при дальнейшем развитии фреймворка.

Режим проверки: документация и код. Для всей серии `KL` нет одного исходного проектного документа, поэтому в качестве живого описания системы использованы инструкции по запуску, README, манифесты, тесты, публичные точки входа и текущая структура модулей.

База сравнения: `origin/main...HEAD` на ветке `kernel_clf_reg`.

## Оценочная карта ML System Design

**Итог:** можно вливать с замечаниями. Тип работы: сопровождение и развитие исследовательско-инженерного слоя для бенчмарков классификации, регрессии и прогнозирования временных рядов. Риск: средний.

**Критические замечания:** нет.

**Авторская оценка:** основные пути стали проверяемыми и воспроизводимыми, но качество сопровождения будет зависеть от того, насколько аккуратно команда будет обновлять документацию, манифесты, внешний архив данных и витрины результатов.

## Краткое резюме

Серия KL переводит IndustrialTS из состояния «много полезных, но разрозненных скриптов и локальных артефактов» в более явную ML-систему.

Главные изменения:

- `benchmark.industrial` стал основным исполняемым слоем для бенчмарков.
- `fedot_ind.core.kernel_learning` стал публичной подсистемой Kernel Learning.
- Эксперименты по бенчмаркам теперь задаются через `BenchmarkSuiteConfig`, `DatasetSpec`, `ModelSpec`, `RunSpec` и `ArtifactSpec`.
- Анализ результатов вынесен из ноутбуков и ручной обработки CSV в общие модули агрегации и визуализации.
- В `examples` появились понятные роли: утилиты текущего API, прикладные сценарии, инструменты для внешних агентов и единая папка артефактов.
- Исторические результаты больше не выглядят как исполняемый код. Они индексируются как справочные данные через манифесты и витрины результатов.

После финальной cleanup-правки основные замечания перед merge закрыты на уровне документации, манифестов и маленьких совместимых изменений кода. Оставшиеся риски относятся к сопровождению: нужно поддерживать актуальность внешнего архива, дат обновления и публичных сравнений.

## Что Изменилось В Работе Фреймворка

### 1. Запуск бенчмарков стал типизированным

Новые запуски должны сводиться к схеме:

```python
typed_config.build_suite_config() -> run_registered_suite(...)
```

`BenchmarkSuiteConfig` теперь несет тип задачи, данные, модели, метрики, параметры запуска и правила сохранения артефактов как один проверяемый объект.

Практический эффект:

- использовать `ModelSpec` и `DatasetSpec` вместо ручных словарей параметров;
- добавлять новые наборы настроек через JSON рядом с тематическим модулем;
- использовать манифесты для воспроизводимых запусков;
- запускать сохранение через `run_registered_suite` или `run_registered_manifest_path`.

### 2. Стало ясно, где живет код бенчмарков

`benchmark.industrial` теперь единственный правильный исполняемый пакет для нового кода бенчмарков. Старый `benchmarking_utils` удален, а прежний подход `benchmark/v2` переведен в текущую структуру.

Практический эффект:

- новый исполняемый код должен жить в тематическом подпакете `benchmark.industrial`;
- `__init__.py` должен оставаться коротким публичным указателем;
- старые корневые классы допустимы только как совместимые обертки;
- исторические пути `v2_kernel_learning` являются источниками данных, а не импортируемыми модулями.

### 3. Результаты запусков стали полноценными входными данными анализа

Запуски теперь сохраняют достаточно информации, чтобы пересобрать таблицы без повторного обучения:

```text
records/
  runs.jsonl
  metrics.jsonl
  predictions.jsonl
  kernel_diagnostics.jsonl
  kernel_selection.jsonl
aggregate/
  runs.csv
  metrics.csv
  predictions.csv
  leaderboard.csv
  run_metadata.json
  summary.md
resolved_config.json
resolved_manifest.json
run_summary.json
artifact_manifest.json
```

Практический эффект:

- ошибки и пропущенные запуски остаются видимыми;
- основные таблицы используют успешные строки метрик, если отчет явно не посвящен ошибкам;
- покрытие, отсутствующие наборы данных, метаданные источников, диагностику, изменение параметров и приросты можно пересобрать единым способом.

### 4. Kernel Learning стал отдельной подсистемой

Основные части:

- `contracts.py`: тип задачи, нормализация, исправление PSD, аппроксимация, набор признаков, набор ядер, отчет о выборе;
- `estimators/`: `KernelEnsembleClassifier`, `KernelEnsembleRegressor`, `KernelEnsembleForecaster`;
- `generators/`: базовые контракты, адаптеры репозитория, легкие генераторы, спецификации операций, реестр;
- `kernels/`: сборка матриц ядер, оценка сложности, аппроксимация Нистрема;
- `selection/`: адаптивный выбор весов ядер, целевые ядра, отчеты о важности;
- `integration/`: определение задачи теплого старта и сборка начальных популяций;
- `cache.py`: кеш ядер по отпечаткам массива и конфигурации.

Практический эффект:

- прямые пользователи импортируют оцениватели из `fedot_ind.core.kernel_learning`;
- бенчмарки получают те же оцениватели через адаптеры моделей `benchmark.industrial`;
- генераторы расширяются через реестр и спецификации операций;
- второй этап эволюционной оптимизации может использовать выбранные генераторы как предположения теплого старта.

### 5. Генераторы признаков разложены по тематическим модулям

Текущая структура:

```text
fedot_ind/core/kernel_learning/generators/
  base.py
  repository.py
  lightweight.py
  specs.py
  registry.py
  adapters.py
```

`adapters.py` теперь является фасадом совместимости, а не единственным огромным файлом с реализациями.

### 6. Витрина результатов стала публичным окном состояния моделей

`benchmark/results/showcase` отвечает на вопрос: «Как сейчас выглядят модели Industrial по сравнению с SoTA и предыдущими версиями Industrial?»

Витрина индексирует:

- UCR/UEA univariate classification;
- multivariate classification;
- TSER regression;
- M4 monthly forecasting.

Для M4 forecasting теперь явно указано: это внутреннее сравнение Industrial до добавления публичного SoTA-источника.

### 7. Примеры переорганизованы по роли

```text
examples/
  utils/                  утилиты текущего API и маленькие фикстуры
  tools_example/          сервисный слой для MCP и внешних агентов
  real_world_examples/    бенчмарковые и прикладные сценарии
  artifacts/              каталог артефактов, облачный пакет, локальная витрина
```

## Как Работать С Обновленным Репозиторием

### Добавить новое направление бенчмарка

1. Добавить исполняемую логику в `benchmark/industrial/<domain>`.
2. Добавить или переиспользовать `BenchmarkSuiteConfig`.
3. Положить таблицы моделей и настройки по умолчанию в JSON рядом с тематическим пакетом.
4. Добавить тонкий запуск в `benchmark/experiments/<family>/<task>`.
5. Сохранять результаты через `run_registered_suite` или помощники реестра манифестов.
6. Подключить агрегацию и визуализацию через общие функции `benchmark.industrial`.
7. Добавить источник в `benchmark/results/showcase/showcase_manifest.json`, если результат должен стать публичным доказательством.
8. Добавить тесты на нормализацию конфигурации, проверку манифеста, агрегацию и короткий проверочный запуск.

### Добавить новый генератор Kernel Learning

1. Писать общий контракт в `generators/base.py` только если он действительно нужен нескольким генераторам.
2. Класть адаптеры репозитория или изменения политики бюджета в `generators/repository.py`.
3. Класть легкие генераторы в `generators/lightweight.py`.
4. Описывать спецификации операций и параметры по умолчанию в `generators/specs.py`.
5. Регистрировать публичное имя генератора в `generators/registry.py`.
6. Держать `generators/adapters.py` только как фасад совместимости.
7. Добавлять тесты в `tests/unit/core/kernel_learning/test_generators.py`.

### Добавить новый пример

1. Сначала выбрать роль:
   - маленький воспроизводимый пример: `examples/utils/current_api`;
   - инструмент для внешнего агента: `examples/tools_example`;
   - прикладной сценарий: `examples/real_world_examples`;
   - сгенерированные отчетные артефакты: `examples/artifacts`.
2. Использовать типизированные конфигурации бенчмарков или актуальные оцениватели Kernel Learning.
3. Держать каталогоподобные константы и настройки в JSON рядом с примером.
4. Не коммитить сырые локальные наборы данных и чекпойнты.
5. В README описывать локальные входы и ожидаемые выходы.
6. Если пример создает переиспользуемые графики или таблицы, регистрировать их в каталоге артефактов.
