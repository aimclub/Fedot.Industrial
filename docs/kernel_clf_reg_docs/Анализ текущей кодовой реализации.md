## 3. Анализ текущей кодовой реализации

### 3.1. Что уже реализовано

`KernelEnsembler` объявлен как `BaseExtractor` и описан как kernel-based ensemble для feature extraction and
classification, с поддержкой one-stage/two-stage kernel learning и multiclass
classification: `fedot_ind/core/ensemble/kernel_ensemble.py:24-29`.

В `__init__` задаются:

- `distance_metric`;
- `kernel_strategy`;
- `learning_strategy`;
- `head_model`;
- список `feature_extractor`;
- learning rate, patience, epoch;
- словарь one/two-stage algorithms: `kernel_ensemble.py:42-58`.

Доступные MKL-алгоритмы берутся
из `KERNEL_ALGO`: `one_step_heur`, `two_step_rmkl`, `two_step_memo`, `one_step_cka`, `one_step_pwmk`: `fedot_ind/core/repository/constanst_repository.py:123-130`.

Активный список baseline feature generators сейчас такой:

- `quantile_extractor`;
- `wavelet_extractor`;
- `fourier_extractor`;
- `eigen_extractor`.

Они строятся через `PipelineBuilder`, причем wavelet/fourier/eigen сначала применяют basis operation,
затем `quantile_extractor`: `constanst_repository.py:131-140`.

При этом `topological_extractor` есть в `KERNEL_BASELINE_NODE_LIST`, но закомментирован
в `KERNEL_BASELINE_FEATURE_GENERATORS`: `constanst_repository.py:131-147`. Это важно: один из четырех ключевых
генераторов главы 2 фактически не участвует в текущем baseline kernel ensemble.

### 3.2. Как работает `KernelEnsembler`

Текущая схема:

1. Проверяется число классов; для multiclass выбирается `ova`, для binary — `ovr`: `kernel_ensemble.py:76-83`.
2. Для каждого feature extractor строится pipeline и
   вызывается `model.fit(input_data).predict`: `kernel_ensemble.py:169-172`.
3. Полученные feature tensors flatten-ятся: `kernel_ensemble.py:173-177`.
4. Для каждого feature matrix считается `squareform(pdist(...))`: `kernel_ensemble.py:178-179`.
5. Эти matrices передаются в MKLpy algorithm: `kernel_ensemble.py:182-185`, `kernel_ensemble.py:188-204`.
6. По весам выбираются лучшие генераторы; жестко берутся top-2 по frequency of class
   assignment: `kernel_ensemble.py:85-105`.
7. Для выбранных генераторов создаются FEDOT pipelines: basis → generator → head model: `kernel_ensemble.py:124-132`.
8. Результат сохраняется как `self.predict = (kernel_ensemble, kernel_data)`: `kernel_ensemble.py:146-167`.

Затем `IndustrialStrategy` вызывает:

```
kernel_ensemble, kernel_data = self.kernel_ensembler.transform(input_data).predictself.solver = self._finetune_loop(kernel_ensemble, kernel_data)
```

и донастраивает каждый созданный pipeline через FEDOT tuner: `fedot_ind/api/utils/industrial_strategy.py:194-211`.

### 3.3. Отдельный OKHS/RKBS слой

В кодовой базе есть отдельный блок `UnifiedOKHSAutoML`, который заявлен как wrapper для OKHS/RKBS workflows across
forecasting, classification, regression: `fedot_ind/core/models/automl/kernel_automl.py:23-27`.

Он строит grid по параметру q для `OccupationKernel`: `kernel_automl.py:101-105`; classification
использует `RKBSCompositeClassifier` с L1 penalty: `kernel_automl.py:114-126`; forecasting выбирает qqq по holdout score
и затем обучает `OKHSForecaster`: `kernel_automl.py:128-164`.

`OKHSForecaster` — отдельная реализация прогнозирования через Occupation Kernel Hilbert Spaces и fractional Liouville
operators: `fedot_ind/core/models/kernel/okhs_forecasting.py:100-137`. Внутри fit строится OKHS fit plan, выбираются
trajectories, resolved window size и qqq, затем либо DMD-модель, либо direct OKHS: `okhs_forecasting.py:185-246`.

Это перспективный блок, но сейчас он **архитектурно не соединен** с `KernelEnsembler` из главы 3. `KernelEnsembler`
занимается feature-generator/MKL selection, а OKHS — отдельным kernel forecaster.

## 4. Основные проблемы реализации и варианты решения

### Проблема 1. В `KernelEnsembler` считаются расстояния, но используются как kernel/Gram matrices

**Описание.** В `generate_grammian` используется `squareform(pdist(...))`, то есть матрица расстояний, а не валидная PSD
kernel matrix: `kernel_ensemble.py:178-179`. MKL-алгоритмы обычно ожидают similarity/Gram matrices. Distance matrix не
обязана быть положительно полуопределенной и может нарушать предпосылки CKA/RMKL/MEMO.

**Почему важно.** Это может делать веса ядер математически некорректными и нестабильными. Особенно опасно на малых
датасетах UCR/UEA: веса будут выглядеть осмысленно, но фактически отражать артефакты метрики.

**Рекомендация:** Конвертировать расстояния в PSD kernels: RBF/Laplacian/cosine kernel + centering/normalization + PSD
check. Нужен отдельный `KernelMatrixBuilder`, который строит train-train и test-train kernels и валидирует PSD.

### Проблема 2. Нет train/test kernel contract

**Описание.** Сейчас `generate_grammian(input_data)` считает только train-train матрицы. Нет сохранения fitted
generators и нет метода, который строит $K(X_{\text{test}}, X_{\text{train}})$. В итоге `KernelEnsembler` не является
полноценным estimator.

**Почему важно.** Для production-модели нужен воспроизводимый `fit/predict`: обучили генераторы на train, сохранили
состояние, для test построили совместимые признаки и ядра.

**Рекомендация:** реализовать полноценный estimator, но постепенно: сначала `KernelFeatureGeneratorRegistry`
и `KernelMatrixBuilder`, затем task-specific estimators (Реализовать
полноценный `KernelEnsembleClassifier/Regressor/Forecaster`)

### Проблема 3. Адаптивность числа генераторов сейчас заменена hard-coded top-2

**Описание.** `_select_top_feature_generators` выбирает `head(2)` по частоте генератора, который максимален для
класса: `kernel_ensemble.py:85-105`.

**Почему важно.** В диссертации речь идет об адаптивном подборе числа моделей признакового пространства через MKL, а не
о жестком выборе двух генераторов. Для разных задач оптимальное число активных генераторов может быть 1, 3, 5 или
class/horizon-specific.

**Рекомендация:** sparse MKL + thresholding по стабильности весов на folds. `top_n` можно оставить как fallback.

### Проблема 4. Нет регуляризатора сложности ядра из главы 3

**Описание.** В главе 3 вводится $C(K)=\|K\|_T/\|K\|_F$​ как более дешевая мера сложности ядра, чем ранг, и как
добавочное слагаемое в лосс. В коде `KernelEnsembler` использует готовые MKLpy алгоритмы, но не реализует этот
регуляризатор.

**Почему важно.** Без контроля сложности система склонна выбирать выразительные, но переобучающиеся генераторы. Это
особенно критично для топологических и высокоразмерных spectral/shapelet/embedding генераторов.

**Рекомендация:** Реализовать собственный objective: alignment - complexity - redundancy - sparsity.

### Проблема 5. Неполный состав генераторов относительно диссертации

**Описание.** Активны quantile/statistical, wavelet, Fourier, eigen/SSA-like; `topological_extractor` есть только в node
list, но выключен из baseline generator registry: `constanst_repository.py:131-147`.

**Почему важно.** Топологический генератор — одна из центральных идей главы 2, особенно для промышленной классификации.
Его отсутствие снижает научную преемственность между диссертацией и кодом.

**Рекомендация:** Ввести multi-fidelity topology: cheap TDA → full TDA только для кандидатов

### Проблема 6. `KernelEnsembler` нарушает ожидаемый контракт `BaseExtractor`

**Описание.** `_transform` возвращает не feature array, а tuple `(kernel_ensemble, kernel_data)`, записанный
в `self.predict`: `kernel_ensemble.py:146-167`. Для `BaseExtractor` это нетипично: обычный extractor возвращает массив
признаков, что видно по базовой логике `_transform`: `fedot_ind/core/models/base_extractor.py:80-118`.

**Почему важно.** Это делает компонент хрупким: он работает только в специальном `IndustrialStrategy`, но плохо
переиспользуется как обычная операция FEDOT/Fedot.Industrial.

**Рекомендация:** разделить обязанности: extractor возвращает features/kernels, strategy builder создает FEDOT
pipelines.

### Проблема 7. Небезопасное remapping поведение по классам

**Описание.** `_create_kernel_data` сравнивает `train_fold.target == classes_described_by_generator[gen]`, где справа
может быть список классов: `kernel_ensemble.py:110-122`. Для multiclass корректнее использовать `np.isin`. Кроме того,
в `IndustrialStrategy._check_predictions` для “missed classes” используется случайный label
fallback: `industrial_strategy.py:313-329`.

**Почему важно.** Случайная разметка пропущенного класса в probability mapping снижает воспроизводимость и может
создавать трудноуловимые ошибки в multiclass classification.

**Рекомендация:** немедленно убрать random fallback; для “other” класса использовать явное calibrated probability или
abstention.

### Проблема 8. `kernel_strategy` имеет опечатку в ключе параметров

**Описание.** В `__init__` используется `params.get('kernel_strategy ', 'one_step_cka')` с пробелом в конце
ключа: `kernel_ensemble.py:42-46`.

**Почему важно.** Передача `kernel_strategy` пользователем, вероятно, игнорируется.

**Рекомендация:** Исправить ключ и добавить backward compatibility для старого ключа с пробелом. Исправить сразу в
первом PR.

### Проблема 9. OKHS/RKBS слой не соединен с `kernel_ensemble`

**Описание.** `UnifiedOKHSAutoML` строит отдельные OccupationKernel/RKBS/OKHS
workflows: `kernel_automl.py:23-27`, `kernel_automl.py:101-126`, `kernel_automl.py:128-176`. Но это отдельный контур, не
связанный с генераторами из главы 2.

**Почему важно.** В коде уже есть потенциально сильный kernel/operator-learning блок. Его стоит использовать как один из
kernel generators или forecasting heads, а не держать изолированно.

| Вариант                                                 | Трудозатраты | Риск    | Отдача        | Поддержка |
|---------------------------------------------------------|--------------|---------|---------------|-----------|
| Оставить отдельным экспериментом                        | низкие       | низкий  | средняя       | низкая    |
| Подключить OKHS как forecasting head                    | средние      | средний | высокая       | средняя   |
| Подключить OKHS как kernel generator + forecasting head | высокие      | средний | очень высокая | средняя   |

**Рекомендация:** сначала OKHS as forecasting head, затем OKHS-derived kernel as generator.
