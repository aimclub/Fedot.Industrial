# OKHS/DMD Refactor Recap

## Контекст

Исходный OKHS/DMD контур в проекте был рабочим на уровне исследовательского прототипа,
но имел несколько системных проблем:

- example-слой содержал утечку данных между train и test;
- plotting, diagnostics и inference были смешаны;
- стратегии выч/методов были скрыты внутри core-методов;
- numpy и torch forecasting wrappers расходились;
- benchmark-модуль был устаревшим и плохо приспособленным для reproducible paper-grade evaluation;
- DMD path на реальных M4-кейсах часто давал либо boundary jump, либо чрезмерно сглаженный прогноз.

## Что уже изменено

### 1. Example stabilization

- `okhs_advanced.py` был превращён в более честный experiment runner.
- Добавлен явный holdout split.
- Вынесены generation/split/evaluation/plotting responsibilities.
- Убрана train/test leakage в examples.

### 2. Typed core policies

- В core OKHS/DMD появились явные policies для numerical stability и regularization.
- `predict` был отделён от plotting/diagnostics.
- Введены shape- и feasibility-checks для key numerical steps.

### 3. Unified forecasting API

- Введён единый vocabulary вокруг `OKHSMethod`.
- Выровнены numpy и torch wrappers.
- `q_policy` стал явной policy, а не скрытой эвристикой.
- Появился benchmark-friendly `get_optimization_info()`.

### 4. Benchmark v2 / product layer

- Построен новый `benchmark/v2` слой с forecasting-first delivery.
- Добавлены adapters для M4 и Monash, затем TSC и TSER.
- Реализованы analytics, publication pack, manifests, registry и run comparison.
- В benchmark artifacts появились series-level overlays, boundary zoom и richer diagnostics.

### 5. OKHS performance optimization

- Ускорено построение Gram-матрицы через precompute quadrature values.
- Добавлен batch-kernel path.
- Добавлено blockwise exact computation для Gram и Liouville matrices.
- Добавлен `tqdm`-мониторинг для Gram/Liouville stages.

### 6. Diagnostics, visualization and boundary fixes

- В benchmark появились series-level plots для history/forecast comparison.
- Добавлены OKHS diagnostics artifacts.
- Исправлен boundary jump через alignment policy.
- Затем hard offset был заменён более мягким tapered boundary correction.

### 7. Adaptive windowing and trajectory preprocessing

- Введён adaptive window policy.
- Добавлены `adaptive_heuristic` и `adaptive_cycle_aware`.
- Появился preprocessing layer для trajectory sampling и low-rank regularization.
- Добавлены diagnostics по окну, stride, selected rank и compression.

## Какие гипотезы проверялись

### Слишком маленькое окно

Гипотеза: окно порядка `1%` длины ряда делает модель слишком локальной и усиливает redundancy.

Результат:

- подтверждено, что окно `8` на ряде длины `700` слишком мало;
- введён adaptive window policy с диапазоном `10%-25%`.

### Избыточный overlap траекторий

Гипотеза: dense sliding windows со stride `1` дают слишком много почти одинаковых trajectories.

Результат:

- подтверждено, что overlap чрезмерный;
- введён adaptive stride и diagnostics по effective trajectory count.

### Медленный Gram/Liouville

Гипотеза: основное bottleneck в Python loops и повторной интерполяции.

Результат:

- подтверждено;
- реализованы caching, batch-kernel и blockwise exact computation.

### Слишком много prediction modes

Гипотеза: после fit в прогноз идёт слишком много spectral modes, и это размывает локальную динамику.

Результат:

- частично подтверждено;
- adaptive prediction mode selector уменьшил число используемых modes;
- однако это не устранило экспоненциальную сглаженность полностью.

### Boundary jump

Гипотеза: predict-path порождает разрыв между last observed point и first forecast point.

Результат:

- подтверждено;
- сначала введён offset-based alignment, затем tapered boundary correction;
- jump был убран, но smoothing сохранился.

### Слишком агрессивная low-rank regularization

Гипотеза: `reconstructed` preprocessing через `U_r S_r V_r^T` сглаживает trajectory matrix слишком сильно.

Результат:

- подтверждено частично;
- rank floor и менее агрессивный stride улучшили ситуацию, но полностью не решили проблему;
- возникла новая рабочая гипотеза про `projected` trajectory representation.

## Что подтвердилось

- Boundary jump был реальной проблемой и действительно устранялся alignment policy.
- Prediction mode selector полезен, но не является достаточным решением.
- Preprocessing траекторной матрицы влияет на качество прогноза сильнее, чем первоначально казалось.
- Текущий `reconstructed` путь с rank-truncated reconstruction, вероятно, создаёт двойное сглаживание:
  сначала при `U_r S_r V_r^T`, затем снова внутри spectral forecast path.

## Что не решено

Главная открытая проблема остаётся той же:

- прогноз OKHS DMD на части M4-рядов всё ещё выглядит как слишком гладкая затухающая экспонента;
- модель недостаточно хорошо держит локальный наклон и локальные паттерны у начала forecast horizon;
- проблема сохраняется даже после fixes для boundary jump и более интеллектуального отбора prediction modes.

## Текущее состояние системы

На текущем этапе в OKHS DMD path уже есть:

- adaptive window selection;
- adaptive stride for trajectory sampling;
- rank regularization с rank floor;
- energy-based mode selection на fit stage;
- adaptive-tail prediction mode selection;
- tapered boundary correction;
- benchmark diagnostics и series-level visualizations.

Эти уже значительно лучше исходного состояния,
но они больше не считаются достаточными для проблемных M4-кейсов.
Основной следующий шаг — переход от `reconstructed trajectory matrix` к `projected trajectory representation`.
