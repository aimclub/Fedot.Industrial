# Transition To Projected Trajectory Representation In OKHS DMD

[Image](docs/img/Create_a_single_wide_6e3ca24d-3173-48bb-872c-ad6b2a94a2e5.png)

## Problem

The earlier adaptive preprocessing for the OKHS DMD path used:

`X -> SVD -> X_hat = U_r S_r V_r^T -> DMD(X_hat)`

where `X` is the trajectory matrix and `X_hat` is its rank-truncated reconstruction.

In practice this created two unwanted effects:

- local patterns were smoothed before the spectral forecasting stage;
- on several M4 series the final forecast still looked like a damped exponential even after fixing the boundary jump and
  improving prediction-mode selection.

## Core Hypothesis

Instead of forecasting on reconstructed windows, the model should forecast on a projected latent representation:

- select rank `r`;
- compute latent states `Z = U_r S_r`;
- build multivariate latent trajectories from `Z`;
- fit OKHS/FractionalDMD on those latent trajectories;
- decode predictions back through `V_r^T`.

The intended effect is:

- less double smoothing;
- better preservation of local dynamics;
- a smaller and better-conditioned DMD problem;
- clearer diagnostics for where smoothing is introduced.

## Representation Policies

The preprocessing layer now distinguishes:

- `trajectory_representation_policy = "none" | "reconstructed" | "projected"`

Meaning:

- `none`
  DMD is trained on sampled original trajectories.
- `reconstructed`
  DMD is trained on `U_r S_r V_r^T`.
- `projected`
  DMD is trained on latent multivariate trajectories built from the latent state sequence `Z = U_r S_r`.

## Current Default Behavior

Current default for adaptive DMD:

- `trajectory_representation_policy = "projected"`

For fixed-window and non-DMD paths the legacy behavior is preserved.

## Implemented Data Flow

### 1. Build the original trajectory matrix

From the source series the pipeline first builds the original trajectory matrix `X`.

Before projection this stage already includes:

- adaptive window resolution;
- original-space trajectory stride;
- rank selection and rank floor;
- trajectory preprocessing diagnostics.

### 2. Compute the low-rank latent basis

For the sampled matrix:

`X = U S V^T`

After rank selection the pipeline keeps:

- `Z = U_r S_r` as the latent state sequence;
- `V_r^T` as the decode basis;
- rank diagnostics such as selected rank, raw selected rank, and rank floor application.

### 3. Build multivariate latent trajectories

This is the most important new stage.

`Z` is treated as a time-indexed sequence of latent states, not as the final training matrix.
The model then builds overlapping latent windows from `Z`, so the effective DMD training tensor has shape:

- `N` = number of latent trajectories;
- `L` = latent window length;
- `r` = latent dimensionality.

So the projected training tensor is:

`latent_trajectories.shape = (N, L, r)`

This is why projected OKHS DMD no longer trains on a 2D matrix of scalar windows. It trains on a collection of
multivariate latent trajectories, where each time step inside a trajectory is itself a latent vector.

### 4. Apply latent trajectory thinning

The projected path now also includes a second thinning stage:

- `latent_trajectory_stride_policy = "dense" | "adaptive"`
- `latent_trajectory_stride`

This stage reduces the number of latent windows `N` without changing rank.
The intent is to lower redundancy in the projected tensor while keeping the latent state sequence itself intact.

Stored diagnostics include:

- `latent_stride`
- `latent_trajectory_stride_policy`
- `dense_latent_trajectory_count`
- `effective_latent_trajectory_count`
- `latent_overlap_ratio`
- `latent_state_count`

### 5. Fit FractionalDMD in latent space

For `trajectory_representation_policy="projected"`:

- `FractionalDMD.fit(...)` receives latent trajectories;
- each latent trajectory is multivariate;
- the spectral model is learned in latent coordinates rather than in reconstructed window space.

This fit path still includes:

- OKHS Gram matrix construction;
- Fractional Liouville operator estimation;
- eigenvalue, mode, and coefficient estimation;
- prediction-mode diagnostics and stability-aware filtering.

### 6. Predict in latent space

At prediction time:

- the last observed segment is mapped into latent coordinates;
- the DMD model predicts continuation in latent space;
- prediction-time mode selection is applied in latent coordinates;
- the result is stored as latent prediction diagnostics.

### 7. Decode back to the original window space

The latent prediction is decoded as:

`X_pred ~= Z_pred V_r^T`

Then the forecast target is read out from the decoded window representation.

The current implementation stores:

- `latent_prediction_shape`
- `decoded_prediction_shape`
- `latent_first_prediction_value`
- `decoded_first_prediction_window`
- `decoded_first_prediction_value`
- `decode_reconstruction_error`

## What Has Already Been Implemented

The following parts are already in code:

- `trajectory_representation_policy`
- projected latent-state preprocessing
- projected training tensor construction
- projected path as the default adaptive DMD representation
- decode metadata through `V_r^T`
- latent trajectory thinning through adaptive latent stride
- benchmark-level propagation of projected diagnostics
- numpy and torch vocabulary alignment for the new representation settings

In other words, the projected design is no longer only a hypothesis document. It now describes the active implementation
direction.

## Risks

### 1. Reduced-order path changes the meaning of DMD input

`projected` is not equivalent to `DMD(U_r S_r V_r^T)`.

This is intentional: the goal is to improve forecasting quality, not to preserve the previous preprocessing contract at
all costs.

### 2. Decode can become the new bottleneck

If rank is too small, latent prediction may be stable while decode fails to preserve window-level structure.

That is why `decode_reconstruction_error` remains mandatory.

### 3. Latent thinning can become too aggressive

If `latent_stride` becomes too large, local short-horizon information may be lost again.

The adaptive latent thinning heuristic should therefore remain diagnostics-heavy and benchmark-driven.

## Success Criteria

Main target cases:

- `D364`
- `D377`
- `D378`

Signs of improvement:

- the forecast stops systematically looking like a damped exponential;
- the boundary zoom shows a more realistic local slope at forecast start;
- `first_step_delta` becomes closer to `first_actual_delta`;
- `MAE`, `sMAPE`, and `MASE` improve versus `reconstructed`;
- `decode_reconstruction_error` stays controlled and interpretable;
- `effective_latent_trajectory_count` is reduced relative to the dense projected path without collapsing local dynamics.
