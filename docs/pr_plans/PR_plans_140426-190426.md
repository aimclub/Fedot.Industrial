# План PR на 14.04.26-19.04.26

## Контекст

К моменту старта этого цикла `Phase 2` по forecasting stack по сути закрыт:

- legacy forecasting boundaries существенно сжаты;
- stage vocabulary стабилизирован;
- stage tuning стал runtime-capable;
- `benchmark/v2` уже понимает family-level diagnostics и stage artifacts;
- neural heads получили рабочий bridge в общий forecasting stack.

На этом фоне следующий цикл больше не про “латание интеграций”, а про переход к
**graph-native primitive execution**.

## Цель цикла

Сделать primitive graph реальным execution substrate для forecasting stack, а shell-модели
оставить в роли упаковки, совместимости и sensible defaults.

## Стратегия

Стратегия этого цикла строится по снижению архитектурного риска:

1. Сначала вынимаем самые крупные “архитектурные острова” в primitive-first форму.
2. Затем даём им graph-native execution runtime.
3. После этого переносим tuning и benchmark на уровень graph/family semantics.
4. И только в конце дочищаем compatibility shells.

## PR-стек текущего цикла

### PR-1. Выделение Neural Primitive Heads

Цель:

- перестать рассматривать native neural forecasting как bridge-only слой;
- сделать neural heads первичными runtime-объектами, а bridge оставить как compatibility wrapper.

Что делаем:

- вводим primitive-oriented neural head module;
- переносим туда registry, diagnostics, input/output normalization и execution contract;
- `NeuralForecastHeadBridge` оставляем как тонкий compatibility shell;
- benchmark/runtime/tuning переводим на новый primitive-oriented neural head.

Почему это первый PR:

- это самый чистый способ начать `Phase 3`;
- neural bridge уже стабилен, значит extraction можно делать без сильного риска;
- это создаёт паттерн для последующего извлечения других forecast heads.

Критерий приёмки:

- native neural heads исполняются без опоры на bridge как на source-of-truth;
- bridge остаётся только как public compatibility entrypoint.

Текущий прогресс:

- primitive-oriented source-of-truth уже вынесен в `neural_forecast_head.py`;
- введён typed runtime contract через `NeuralForecastHeadSpec`;
- добавлен общий execution helper с typed run-result, который уже используется benchmark-path;
- benchmark и stage-tuning runtime уже используют factory-path нового primitive head;
- `NeuralForecastHeadBridge` сохранён как compatibility shell с прежним публичным конструктором;
- native neural forecasting operations уже переведены на `ts_model` runtime path и новые implementation classes.

Статус:

- `PR-1` можно считать функционально закрытым.

### PR-2. Завершение OKHS Primitive Graph

Цель:

- довести `okhs` до полноценной primitive-first архитектуры.

Что делаем:

- продолжаем extraction decomposition/rank/prediction orchestration;
- формализуем `okhs_fdmd_head` и direct-kernel head;
- уменьшаем объём orchestration в kernel-level shell.

Почему вторым:

- `okhs` остаётся самым сложным residual-монолитом после neural bridge;
- после PR-1 уже будет понятный паттерн “head-first extraction”.

Критерий приёмки:

- `okhs_fdmd_forecaster` становится тонкой named composition над runtime/primitive слоем;
- compatibility shell перестаёт быть orchestration center.

Текущий прогресс:

- добавлены typed `OKHS` spec и run-result;
- появился общий execution helper для `okhs_fdmd_forecaster`;
- benchmark path уже начал использовать этот helper как новый source-of-truth.

### PR-3. Graph-Native Forecasting Execution Runtime

Цель:

- дать forecasting stack настоящий graph-native execution path.

Что делаем:

- исполняем forecasting graphs из primitive nodes;
- добавляем graph-level diagnostics и optimization metadata;
- вводим graph-level runtime contract для shell/composite моделей.

Почему третьим:

- graph runtime должен опираться на уже extracted heads, а не на bridge-only wrappers;
- иначе мы просто перенесём старую shell-centric проблему на новый слой.

Критерий приёмки:

- хотя бы один нетривиальный forecasting graph исполняется end-to-end без bespoke shell orchestration.

### PR-4. Graph-Aware Tuning and Evaluation

Цель:

- поднять tuning и evaluation с shell-level на graph-level.

Что делаем:

- stage tuning для graph nodes;
- graph-level holdout / rolling-origin evaluation;
- reusable tuning reports для benchmark и developer workflows.

Почему четвёртым:

- сначала нужен graph execution runtime;
- потом уже можно корректно тюнить не shell, а сам graph.

Критерий приёмки:

- tuned primitive graphs можно сравнивать напрямую с untuned graphs и shell-baselines.

### PR-5. Family-First Benchmark and Routing Consolidation

Цель:

- сделать family/graph perspective основным для benchmark и routing.

Что делаем:

- сравниваем graph variants внутри families;
- расширяем family-level routing evaluation;
- делаем shell-name artifacts вторичными по отношению к family/graph artifacts.

Почему пятым:

- benchmark должен строиться на уже стабильной graph semantics.

Критерий приёмки:

- benchmark отвечает не только “какая модель выиграла”, но и:
  - какая family была рекомендована;
  - какая family реально выиграла;
  - какой graph variant внутри family оказался лучшим.

### PR-6. Cleanup Compatibility Packaging

Цель:

- оставить compatibility shells только как упаковку и public API continuity.

Что делаем:

- убираем дублирующую orchestration logic из bridge/wrapper classes;
- сужаем их до naming/defaults/public compatibility;
- помечаем truly legacy wrappers там, где это уместно.

Почему последним:

- cleanup безопасен только после того, как runtime substrate уже реально сместился на primitive graph.

Критерий приёмки:

- wrappers больше не управляют forecasting runtime semantics.

## Ближайший стартовый PR

Первым в работу берём:

- `PR-1` Выделение Neural Primitive Heads

Почему:

- он уже технически подготовлен через existing neural bridge;
- он даёт быстрый архитектурный выигрыш;
- он создаёт reusable pattern для следующих PR.

## Что не входит в этот цикл

Чтобы цикл не расползался, в него сознательно не включаем:

- полный редизайн внутренних neural architectures;
- тотальную замену всех FEDOT boundary objects;
- model-quality deep dives для всех forecasting families;
- переход на generic builder-first API вместо named shells.

## Ожидаемый результат цикла

К концу цикла forecasting stack должен перейти от
“shell-first architecture with extracted helpers”
к
“primitive-graph architecture with shell packaging”.
