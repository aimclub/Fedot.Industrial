# История разработки forecasting-ветки

## Назначение

Этот документ объясняет, почему forecasting-ветка пришла к текущей архитектуре. Reference-документы отвечают на вопрос “как пользоваться”, а эта история отвечает на вопрос “почему так сделано”.

## 1. Отказ от legacy FEDOT `lagged`

Изначально forecasting path пытался встроить FEDOT `lagged` operation напрямую. На практике это дало слишком много ошибок на стыке Industrial runtime и ядра FEDOT.

Решение ветки:

- добавить Industrial operation `hankelisation`;
- сделать её pipeline-compatible;
- вынести `window_size` и `stride` в явный contract;
- использовать `hankelisation -> ridge` как проверенный baseline.

Итог: Industrial получил собственный управляемый trajectory-transform вместо хрупкого внешнего fallback.

## 2. Stage-first forecasting runtime

Следующий шаг был архитектурным: перестать воспринимать forecasting-модель как монолит.

Целевая схема:

```text
trajectory_transform -> decomposition -> rank_truncation -> forecast_head
```

Эта схема стала общей для:

- lagged models;
- low-rank models;
- HAVOK/OKHS;
- neural heads;
- hybrid ensembles.

## 3. Named composite models

Ветка добавила stage-aware модели:

- `lagged_ridge_forecaster`
- `low_rank_lagged_ridge_forecaster`
- `hybrid_ensemble_forecaster`
- `okhs_fdmd_forecaster`

Главная цель была не заменить все старые entrypoints сразу, а дать новые обертки, которые модули бенчмаркинга и
модуль тюнинга могут понимать одинаково.

## 4. Explicit stage tuning

Stage tuning был вынесен в отдельный слой:

- stage plan;
- stage search spaces;
- sequential stage tuning execution;
- runtime bridge на реальных временных рядах;
- multi-fold temporal split.

Важное решение: stage tuning больше не оценивается на одной "хвостовой отложенной выборке" по умолчанию. Runtime
поддерживает temporal CV через `time_series_split`, `expanding_window`, `rolling_window` и compatibility `holdout`.

## 5. Benchmark/v2 как runtime-facing слой

`benchmark/v2` был усилен как основной слой вычислений:

- `ForecastingSuiteRunner`;
- item-level persistence;
- resume mode;
- post-fit tuning comparison;
- progress/verbosity policy;
- stage diagnostics artifacts;
- family-level summaries.

Это решает практическую боль длинных benchmark-запусков: если процесс падает, уже посчитанные результаты не теряются.

## 6. Result visualization

Для анализа `progress/items/*.json` добавлен отдельный слой визуализации.

Он строит:

- history + forecast-zone zoom;
- fold diagnostics;
- per-series relative gain;
- aggregate baseline vs tuned scatter;
- regime diagnostics summary;
- HTML summary с gallery history-графиков.

Этот модуль специально работает от инкрементальный чекпоинт, чтобы его можно было запускать даже на частично завершённых
benchmark-прогонах.

## 7. Neural head bridge и neural runtime cleanup

Neural forecasting был приведён к общей спецификации.

Сделано:

- `patch_tst_model`, `tst_model`, `tcn_model`, `deepar_model`, `nbeats_model` подключены как `neural_forecaster`;
- defaults вынесены в общий neural runtime layer;
- `epochs`, `batch_size`, `learning_rate`, `device` исключены из search space;
- scheduler и early stopping отвечают за обучение;
- diagnostics стали частью stage-aware contract.

## 8. Реорганизация `ts_forecasting`

Пакет был разложен по тематическим подпакетам:

- `lagged_model`
- `dmd_models`
- `ensemble_models`
- `neural_models`
- `forecast_tuning`
- `regime_utils`

Legacy paths оставлены только как compatibility shims там, где это нужно для старых импортов.

## 9. Текущие ограничения

На момент подготовки ветки к публикации сознательно остаются следующие ограничения:

- не весь старый OKHS backend превращён в полностью graph-native primitive runtime;
- часть legacy examples и локальных результатов остаётся вне PR;
- neural models подключены к stage vocabulary, но не все они одинаково глубоко оптимизированы по качеству;
- full benchmark cohorts требуют отдельного runtime окружения и не должны блокировать merge небольших API/docs/test changes.

## 10. Что переносится на следующий цикл

Следующий цикл стоит вести не как “ещё один большой рефакторинг”, а как серию точечных PR:

- довести OKHS до более полного primitive graph;
- расширить quality gates на реальные M4 cohorts;
- усилить model-specific diagnostics;
- добавить retry-failed-only режим для benchmark resume;
- постепенно сокращать compatibility shims, когда downstream imports стабилизируются.
