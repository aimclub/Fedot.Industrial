# Политика артефактов перед вливанием forecasting-ветки

## Назначение

Этот документ фиксирует, какие файлы из текущей forecasting-ветки должны попадать в PR, а какие остаются локальными результатами экспериментов.

Главное правило: в `main` попадают код, тесты, маленькие runnable examples и документация. Результаты запусков, датасеты, checkpoint-файлы и визуализации не коммитятся.

## Что не включаем в PR

- `benchmark/results/`
- `benchmark/industrial/examples/benchmark/results/`
- `benchmark/industrial/examples/**/visualizations/`
- `results_of_experiments/`
- локальные датасеты в `data/`, `examples/data/`, `fedot_ind/data/`
- checkpoint-файлы и веса моделей: `*.pth`
- архивы: `*.zip`, `*.7z`
- тяжёлые 3D/point-cloud артефакты: `*.stl`, `*.ply`
- generated media из explainability или benchmark-прогонов: `*.gif`

## Что можно включать в PR

- исходный код новых runtime/model/benchmark модулей;
- unit-тесты и маленькие synthetic fixtures;
- runnable scripts без встроенных результатов;
- русскоязычную документацию в `docs/dev`;
- `.gitignore`, если он защищает от случайного добавления локальных артефактов.

## Проверка перед коммитом

1. Выполнить `git status --short`.
2. Убедиться, что в списке нет benchmark output, датасетов, checkpoint-файлов и архивов.
3. Если новый runnable example лежит рядом с output-папкой, добавить только `.py` script, а не `results/`.
4. Если нужно сохранить результаты эксперимента для анализа, оставить их локально или вынести во внешнее хранилище, но не включать в PR.

## Текущий контекст ветки

Ветка содержит много локально сгенерированных файлов от benchmark/debug-прогонов. Они полезны для анализа, но не являются частью publishable change-set. Поэтому `.gitignore` расширен под новые output-пути `benchmark/industrial` и типовые тяжёлые расширения.
